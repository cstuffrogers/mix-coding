# /plan-ceo-review — 创始人策略审查

混合工作流：**上下文加载 → 创始人三问 → 结构化输出**

## Usage

```text
/plan-ceo-review                  # 审查当前 feature 分支
/plan-ceo-review "用户邀请功能"    # 审查指定功能
```

## 执行流程

### Phase 0: 上下文加载

1. **读取 spec/plan** (`step 1`) — 读取功能规格、PRD、任务文档，理解当前方案
2. **recall** (`step 0.5`) — 注入历史产品决策和用户反馈

### Phase 1: 创始人三问

3. **10x 分析** (`step 2`) — "如果只给一周，10x 版本长什么样？不做渐进式修补，从第一性原理重新定义问题"
4. **精简化** (`step 3`) — "什么可以砍？识别 scope creep、镀金需求、过度工程化。最简可上线版本需要哪些？"
5. **用户价值** (`step 4`) — "用户会注意到并关心吗？这个功能移动什么核心指标？如果不上线，用户会流失吗？"

### Phase 2: 结构化输出

6. **三桶分类** (`step 5`)：

| 桶 | 定义 | 行动 |
|------|------|------|
| 🔴 **Must-Do** | 不做用户会流失 | 本周内完成 |
| 🟡 **Nice-to-Have** | 锦上添花但不改变决策 | 可延后 |
| ⚪ **Cut** | 镀金/过度工程/没人会注意 | 砍掉或永久推迟 |

7. **生成报告** (`step 6`) — 每条建议：是什么 + 为什么 + 用户影响 + 建议行动
8. **remember** → **consolidate** → **notify**

### 审查原则

- 从用户视角出发，不讨论实现细节
- 挑战一切假设：这个功能真的需要存在吗？
- 偏好激进简化：宁可少做，不堆砌功能
- 10x = 不是好 10%，而是好 10 倍

### 工具链覆盖

| 阶段 | 工具 | 类型 |
|------|------|------|
| 上下文 | spec 文档 + 记忆召回 | Read |
| 分析 | 结构化 LLM 审问 | Skill |
| 输出 | 三桶分类 + 可执行建议 | 对话模式 |
