# Ponytail — 懒人高效模式

> 决策阶梯规则。Agent 在写任何代码之前必须先过这 6 关。
> 与 `karpathy-principles.md` 互补：Karpathy 管"怎么写"，Ponytail 管"要不要写"。
> 来源: https://github.com/DietrichGebert/ponytail (MIT)

---

## 决策阶梯

写代码之前，从第 1 级开始，**命中即停**：

1. **YAGNI** — 这东西真的需要存在吗？不需要 = 不写
2. **标准库** — stdlib 已经有了？直接用
3. **平台原生** — Node/browser 原生 API 能搞定？直接用
4. **已安装依赖** — 项目已装的包能解决？直接用
5. **一行搞定** — 能一行写完就别写两行
6. **最少代码** — 以上都不行，才写最少能用的代码

## 硬规则

- 不要未被要求的抽象层
- 不要新依赖，除非别无选择
- 不要没人要的样板代码
- 优先删减而非堆砌。允许先加后删的重构路径（加过渡层→迁移调用方→删旧代码）。无聊 > 聪明。文件越少越好
- 对复杂需求反问："你真的需要 X 吗，还是 Y 就够了？"
- 两种 stdlib 方案同样简短时，选边界情况正确的那个。懒 = 少写代码，不是更脆的算法
- 有意简化处用 `ponytail:` 注释标记。如果简化有已知天花板（全局锁、O(n²) 扫描、朴素启发式），注释中写明天花板和升级路径

## 不偷懒的事

以下绝不简化，即使"一行能搞定"也不行：

- **用户输入** — 信任边界校验（类型、范围、SQL 注入、XSS）
- **错误处理** — 涉及数据丢失/损坏的场景必须完整处理
- **安全** — 认证、授权、密钥管理、敏感数据
- **无障碍** — `alt`、`aria-label`、焦点管理、键盘可达
- **明确需求** — 用户说了要的功能，完整实现

不适用场景：安全漏洞修复、hotfix、数据恢复脚本 — 这些场景正确性优先于简洁性。

## 测试规则

| 代码类型 | 要求 |
|---------|------|
| 琐碎一行（映射、取值、简单赋值） | 不需要 |
| 纯函数（有明确输入→输出） | N 个分支 → N 个 test case，每个至少 1 个 assert |
| 有分支/状态流转的逻辑 | 最小测试文件（1 个 .test.js，happy path + 关键边界） |
| 调用外部系统（DB/API/文件） | 集成测试 |

单元测试可以 mock 外部调用；集成测试用真东西。不要测试框架、不要 fixtures。

## 与其他规则的冲突优先级

冲突时按此顺序裁决：

| 优先级 | 规则文件 | 裁决原则 |
|--------|---------|---------|
| 1（最高） | `coding.md` + 用户明确要求 | 用户说了算 |
| 2 | `ponytail.md`（本文件） | 能不写就不写，能用现成的就用现成的 |
| 3 | `karpathy-principles.md` | 如果必须写，写得简洁、手术式 |
| 3.5 | `codeguardian.md` | 改完之后验证：物理边界、自动验证、失败回滚、人工门控 |
| 4 | `od-craft/*.md` | 写出来后，过设计规范审查 |
| 5 | `/simplify` `/review` | 写完后，持续清理和审查 |

> 例：Ponytail 说"一行搞定"但不需要 try-catch。Karpathy 说"不写不可能发生的异常处理"。一致，无冲突。
> 
> 例：Ponytail 说"不要新依赖"。但用户明确要装某个库。用户说了算（优先级 1 > 2）。

## 与现有规则的关系

| 规则 | 职责 |
|------|------|
| 本文件 | **写之前**阻断：这东西需要写吗？用现成的行不行？ |
| `karpathy-principles.md` | **写之时**约束：简洁、手术式、目标驱动 |
| `codeguardian.md` | **改之后**验证：物理边界、自动验证、失败回滚、人工门控 |
| `/simplify` 工作流 | **写之后**清理：识别并删除冗余 |
| `od-craft/anti-ai-slop.md` | **审查时**检测：AI 塑料感模式 |
