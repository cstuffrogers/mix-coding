# 核心编码规则

> Ponytail + Karpathy + CodeGuardian 三合一。写代码前中后的完整约束。

## 写之前：决策阶梯

从第 1 级开始，**命中即停**：

1. **YAGNI** — 这东西真的需要存在吗？不需要 = 不写
2. **标准库** — stdlib 已经有了？直接用
3. **平台原生** — Node/browser 原生 API 能搞定？直接用
4. **已安装依赖** — 项目已装的包能解决？直接用
5. **一行搞定** — 能一行写完就别写两行
6. **最少代码** — 以上都不行，才写最少能用的代码

优先删减而非堆砌。文件越少越好。无聊 > 聪明。

## 写之时：四大约束

1. **先想再写** — 不确定时主动问，有歧义时呈现多种解释，有更简单方案就说出来
2. **简洁优先** — 不实现没要求的功能，不为单次使用建抽象层，不为不可能场景加错误处理
3. **手术式修改** — 只改必须改的，不顺手重构相邻代码，不改注释格式。只删自己造成的孤儿代码
4. **目标驱动** — 定义成功标准再动手："先写复现测试，再让它通过"

## 改之后：物理边界

| 边界 | 值 | 触发条件 |
|------|-----|---------|
| 行数变化 | ≤ ±50 行 | 超出 → 驳回 |
| 新依赖 | 0 | 新增 import → 驳回 |
| 复杂度增长 | ≤ 130% | 超出 → 驳回 |
| 测试/基准/质量 | 全部通过 | 任一失败 → 自动回滚 |
| 人工审查 | 必须 | AI 不可自行合并 |

配置：`.codeguardian.yaml`。MCP 工具：`codeguardian_optimize`/`codeguardian_status`。

## 绝不简化的事

- **用户输入** — 类型/范围/SQL注入/XSS校验
- **错误处理** — 涉及数据丢失/损坏必须完整
- **安全** — 认证/授权/密钥管理/敏感数据
- **无障碍** — alt/aria-label/焦点管理/键盘可达
- **明确需求** — 用户要求的功能完整实现

## 优先级

| 优先级 | 规则 | 职责 |
|--------|------|------|
| 1 | `coding.md` + 用户要求 | 用户说了算 |
| 2 | 本文件 | 写之前决策 → 写之时约束 → 改之后验证 |
| 3 | `od-craft/*.md` | 设计规范审查 |
| 4 | `/simplify` `/review` | 清理/审计 |

## 不适用场景

安全漏洞修复 / hotfix / 数据恢复 — 正确性优先于简洁性。
