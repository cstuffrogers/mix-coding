# Karpathy 编码四诫

> 基于 Andrej Karpathy 对 LLM 编程行为的观察蒸馏出的 4 条编码原则。
> 来源: https://github.com/multica-ai/andrej-karpathy-skills
>
> 这些原则直接回应 Agent 最常见的三类错误：
> (1) 不检查假设就执行、(2) 过度工程化、(3) 修改与任务无关的代码。

---

## 原则 1：先思考再编码（Think Before Coding）

**不要假设。不要隐藏困惑。明确呈现权衡。**

Agent 常常静默选择一个解释然后一条路走到黑。本原则强制显式推理：

- **显式声明假设** — 不确定时，主动询问而非猜测
- **呈现多种解释** — 当存在歧义时，不静默选择
- **在应该时推回** — 如果存在更简单的方法，说出来
- **困惑时停止** — 明确指出不清楚的地方并请求澄清

**反模式**：Agent 看到 "add auth"，假设需要 OAuth 2.0 + JWT + refresh token，动手编写完整认证系统。用户其实只想要 HTTP Basic Auth。

---

## 原则 2：简洁优先（Simplicity First）

**解决问题的最少代码。不添加推测性内容。**

对抗过度工程的倾向：

- 不实现未被要求的功能
- 不为单次使用创建抽象层
- 不添加未被要求的"灵活性"或"可配置性"
- 不为不可能发生的场景添加错误处理
- 如果 200 行可以缩到 50 行，重写它

**测试标准**：资深工程师会说"这过度复杂了"吗？如果是，简化它。

**与本系统 /simplify 场景的关系**：`/simplify` 场景正是此原则的系统化实现——范围确认→基线测试→识别简化机会→选择方向→增量测试→审查验证。

---

## 原则 3：手术式修改（Surgical Changes）

**只碰必须改的。只清理自己造成的混乱。**

编辑已有代码时：

- 不"改进"相邻代码、注释或格式
- 不重构没有坏的东西
- 匹配已有风格，即使你更倾向于另一种写法
- 如果注意到不相关的死代码，提出来——不删除它

当你的修改造成孤儿代码时：

- 删除被**你的修改**变成未使用的 imports/变量/函数
- 不删除预先存在的死代码，除非明确要求

**测试标准**：每一行被修改的代码都应直接追溯到用户的需求。

**与本系统场景的关系**：`/simplify` 的步骤 1 强制"确认简化范围：仅修改当前 feature 相关代码"；`AGENTS.md` 的 WIP=1 规则防止跨功能越界。

---

## 原则 4：目标驱动执行（Goal-Driven Execution）

**定义成功标准。循环直到验证通过。**

将命令式任务转化为可验证的目标：

| 不要... | 转化为... |
|---|---|
| "添加验证" | "先为无效输入写测试，然后让它们通过" |
| "修复这个 bug" | "先写一个复现它的测试，然后让它通过" |
| "重构 X" | "确保测试在重构前后都通过" |

对于多步骤任务，先陈述简要计划：

```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强的成功标准让 Agent 能独立循环执行。弱的标准（"让它能用"）需要持续的澄清。

**与本系统场景的关系**：`/bugfix` 的测试基线步骤（"修复前确保可复现失败测试"）和 `/feature` 的基线测试都体现了此原则——先写复现问题的测试，再修改代码，最后验证通过。

---

## 如何判断原则生效

这些原则生效的标志：

- **Diff 中不必要的改动变少** — 只有被请求的改动出现
- **因过度复杂而重写的情况变少** — 第一次就写出简洁的代码
- **澄清问题在实现之前就被提出** — 不是出错之后
- **PR 干净、最小** — 没有顺手重构或"改进"

## 权衡说明

这些原则偏向**谨慎优先于速度**。对于琐碎任务（简单的拼写修正、显然的一行改动），请使用判断力——不是每个改动都需要完整的严格流程。

目标是在非琐碎工作上减少代价高昂的错误，而非拖慢简单任务。
